# Skill: read-mode

## 目的

讓 agent 在面向人類輸出**需要閱讀吸收的說明性資訊**時，改用「分段閱讀 + 互動推進」模式，而不是一次傾倒整篇文字。

## 何時使用

在以下情況主動啟用 read-mode：

- 正在**回答使用者提問**，且答案屬於解釋型 / 教學型 / 分析型內容
- 只要回覆的主要任務是「讓人理解」，即使不是超長文，也應優先考慮啟用
- 內容需要人類逐步理解，例如：
  - 架構解釋
  - 設計權衡
  - 長篇 debug / RCA 說明
  - 規格導讀
  - 教學式分解

## 何時不要使用

以下情況**不要**進入 read-mode：

- tool call / build / test / git / runtime 狀態回報
- 錯誤摘要、驗證結果、commit 結果、diff 結論
- 使用者要求「直接講完 / 一次講完 / 給完整答案」
- 答案本來就很短（1~2 小段可講完）
- 使用者明確要求純列表、純指令、純結論

## 操作規則

1. 先將內容拆成自然段落或章節。
2. 一次只輸出一段，不要一次貼完整長文。
3. **以約 20 行上下為一個閱讀單位**；到達此長度附近就應停下，不必等到整個章節講完。
4. 每段結尾停下，使用 `question` 讓使用者決定下一步。
5. `question` 至少要提供：`是`、`否`，並允許使用者輸入其他意見。
6. 除非使用者要求一次講完，否則不要自動連續輸出後續段落。
7. 若內容中有結論可先講，第一段可先給「總覽 + 本段重點」，但仍須停下等待選擇。

## 建議段落大小

- 每段以 **約 20 行上下** 為主要節奏基準
- 若自然小節不足 20 行，可在小節結尾停下
- 若小節超過 20 行，應優先在 20 行附近找自然停點

## question 選項模板

最少應提供：

1. `是`
2. `否`

並開啟自訂輸入，讓使用者可直接輸入其他意見。

若情境適合，可額外提供：

- `先給摘要`
- `直接看結論`
- `這段有問題`

視內容可加：

- `給例子`
- `看程式層細節`
- `跳到實作建議`

## 啟用判斷 heuristics

若同時滿足以下兩點，應主動考慮啟用：

1. **回答類型是說明性資訊**，而非執行結果回報
2. **人類需要閱讀吸收**，即使只有中等長度也算

簡單判斷句：

> 如果現在這則回覆的主要內容是「幫人理解」，而不是「回報工具結果」，就優先考慮 read-mode；並且每約 20 行就停一次。

## 退出條件

當出現以下任一條件時，退出 read-mode：

- 使用者選擇 `直接看結論`
- 使用者明確說「一次講完」
- 後續回合轉為命令執行 / 測試 / git / runtime 操作

## 目標效果

- 讓人類能以自己的閱讀速度推進資訊
- 減少一口氣輸出多頁說明造成的認知負擔
- 保持 technical explanation 的可讀性與互動性
