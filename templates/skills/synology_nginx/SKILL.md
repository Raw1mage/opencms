# Synology NAS: Custom Nginx Configuration for Reverse Proxy

Synology DSM auto-generates its nginx reverse proxy config. This skill documents two methods to inject custom directives, their tradeoffs, and the practical reality of which one actually works.

## Key Files

| File | Purpose |
|------|---------|
| `/usr/syno/etc/www/ReverseProxy.json` | DSM reverse proxy registry (UUID → domain mapping) |
| `/usr/local/etc/nginx/sites-enabled/server.ReverseProxy.conf` | Auto-generated nginx config (all reverse proxy server blocks) |
| `/usr/local/etc/nginx/conf.d/<UUID>/user.conf` | Persistent per-entry override (Method B — may not work on all DSM versions) |

## Method A: Direct Edit of `server.ReverseProxy.conf` (PROVEN WORKING)

This is the method that has been **verified working** on the current NAS (DSM 7.x). The crm.sob.com.tw WebSocket support was configured this way and has been stable.

### Tradeoff

This file is auto-generated by DSM. Edits **will be overwritten** if you modify any reverse proxy entry through DSM's web UI. However, it is the only method confirmed to work on the current DSM version.

### Step-by-Step

#### 1. Find the target domain in `ReverseProxy.json`

```bash
sudo cat /usr/syno/etc/www/ReverseProxy.json | python3 -m json.tool
```

The JSON is keyed by UUID. Each entry has:
- `frontend.fqdn` — the domain name
- `frontend.port` — listening port (usually 443)
- `backend.fqdn` / `backend.port` — where traffic is proxied to
- `description` — label shown in DSM UI

#### 2. Find the target `location /` block in the generated config

```bash
sudo grep -n "server_name\|proxy_pass\|Upgrade" /usr/local/etc/nginx/sites-enabled/server.ReverseProxy.conf
```

Identify the `server_name <your-domain>` block and its `location /` section.

#### 3. Inject custom directives

Add directives **inside the existing `location /` block**, between the last `proxy_set_header` line and the `proxy_pass` line. Do NOT create a separate `location /` block (it would conflict with the existing one).

Example patch for WebSocket support:

```nginx
        proxy_set_header        X-Forwarded-Proto            $scheme;

        # WebSocket support for OpenCode PTY + SSE
        proxy_set_header        Upgrade             $http_upgrade;
        proxy_set_header        Connection          "upgrade";

        proxy_pass http://192.168.100.10:1080;
```

**Important**: Do NOT add `proxy_read_timeout` if one already exists earlier in the same `location` block — nginx will reject duplicate directives.

#### 4. Test and reload

```bash
sudo nginx -t && sudo nginx -s reload
```

The `conflicting server name` warnings are normal for DSM (multiple `server` blocks with the same names exist across included files) and do not affect functionality.

#### 5. Verify

```bash
sudo nginx -T 2>/dev/null | grep -A30 "server_name <your-domain>"
```

Confirm the Upgrade/Connection headers appear in the output.

### Re-applying after DSM overwrites

If DSM regenerates the config (e.g., after editing any reverse proxy in DSM UI), you'll need to re-apply the patch. To make this easier, keep a record of what was added (this skill serves as that record).

## Method B: `user.conf` in `conf.d/<UUID>/` (DOCUMENTED BUT NOT WORKING ON CURRENT DSM)

This method is widely documented online for Synology DSM. The idea is:

```
/usr/local/etc/nginx/conf.d/<UUID>/user.conf
```

gets included inside the server block via:

```
include /usr/local/etc/nginx/conf.d/<UUID>/*;
```

**However**, on the current NAS, `server.ReverseProxy.conf` does **not** contain any `include conf.d/<UUID>/*` directives. The `user.conf` files exist on disk but are never loaded. This may be DSM-version-specific — newer or older versions might support this mechanism.

### How to check if Method B works on your DSM

```bash
grep "include.*conf.d" /usr/local/etc/nginx/sites-enabled/server.ReverseProxy.conf
```

If this returns nothing, Method B won't work — use Method A instead.

If it returns include lines with UUIDs, Method B is available:

```bash
sudo mkdir -p /usr/local/etc/nginx/conf.d/<UUID>/
# Write directives WITHOUT wrapping in location {} — they're included inside the server block
sudo tee /usr/local/etc/nginx/conf.d/<UUID>/user.conf << 'EOF'
proxy_set_header Upgrade $http_upgrade;
proxy_set_header Connection "upgrade";
EOF
sudo nginx -t && sudo nginx -s reload
```

## Common Recipes

### WebSocket Support

Without this, nginx replaces `Connection: Upgrade` with `Connection: close`, breaking WebSocket handshakes.

```nginx
# Add inside existing location / block (Method A)
proxy_set_header        Upgrade             $http_upgrade;
proxy_set_header        Connection          "upgrade";
```

### SSE (Server-Sent Events) Support

For long-lived HTTP streaming responses:

```nginx
proxy_buffering off;
proxy_cache off;
```

### Extended Timeout for Long-Lived Connections

Only add if there isn't already a `proxy_read_timeout` in the same block:

```nginx
proxy_read_timeout 3600;
```

---

## Why This Matters

DSM's reverse proxy UI only exposes basic settings (backend host/port, HTTPS, HSTS, HTTP/2). It does **not** expose:
- WebSocket header forwarding (`Upgrade` / `Connection`)
- Proxy buffering control
- Custom location blocks or additional headers
