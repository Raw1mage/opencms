Maintain a structured task list for the current session. The list is observability — the user / sidebar reads it to see the model's plan and progress; the runtime is also indirectly driven by it (autonomous-mode continuation reads the in_progress / pending residue).

## Schema

- `content`: brief description of the task.
- `status`: one of `pending` | `in_progress` | `completed` | `cancelled`.
- `priority`: `high` | `medium` | `low`. Defaults to `medium`.
- `id`: stable identifier. Auto-generated if omitted on first write; preserve when updating.

## Parameters

- `mode` (optional): why this update is happening.
  - `status_update` — toggling progress on existing tasks (no structure drift).
  - `plan_materialization` — first time committing a plan to the list.
  - `replan_adoption` — replacing structure due to scope shift.
  - The runtime auto-promotes `status_update` to `working_ledger` when structure actually changed, so the field can be omitted in most cases.
- `todos`: the full updated todo list. Send `content` / `status` / `priority` / `id` only — `action` metadata is inferred server-side.

## Concurrency

Only one task should be `in_progress` at a time.

## Idempotency

A call whose `todos` is byte-equivalent to the current persisted state is a no-op write. The runtime tolerates it but it carries no information.

## Frequency / trigger rules

This is a tool schema. Rules about WHEN to call this tool — frequency, status-change triggers, autonomous continuation, ledger discipline — live in the system prompt (`SYSTEM.md`) and agent driver prompts, not here.
