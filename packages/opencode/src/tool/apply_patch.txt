Use the `apply_patch` tool to edit files. HARD REQUIREMENT: Do NOT call `apply_patch` on an existing file unless your immediately previous action was `read` on that exact file. If the last action was not reading that file, stop and read it first. This is mandatory because stale context causes match failures, bad hunks, and slow retries.

Your patch language is a stripped‑down, file‑oriented diff format designed to be easy to parse and safe to apply. You can think of it as a high‑level envelope:

*** Begin Patch
[ one or more file sections ]
*** End Patch

Within that envelope, you get a sequence of file operations.
You MUST include a header to specify the action you are taking.
Each operation starts with one of three headers:

*** Add File: <path> - create a new file. Every following line is a + line (the initial contents).
*** Delete File: <path> - remove an existing file. Nothing follows.
*** Update File: <path> - patch an existing file in place (optionally with a rename).

May be immediately followed by *** Move to: <new path> if you want to rename the file.
Then one or more "hunks", each introduced by @@ (optionally followed by a hunk header).
Within a hunk each line starts with:

For instructions on [context_before] and [context_after]:
- By default, show 3 lines of code immediately above and 3 lines immediately below each change. If a change is within 3 lines of a previous change, do NOT duplicate the first change's [context_after] lines in the second change's [context_before] lines.
- If 3 lines of context is insufficient to uniquely identify the snippet of code within the file, use the @@ operator to indicate the class or function to which the snippet belongs. For instance, we might have:
@@ class BaseClass
[3 lines of pre-context]
- [old_code]
+ [new_code]
[3 lines of post-context]

- If a code block is repeated so many times in a class or function such that even a single `@@` statement and 3 lines of context cannot uniquely identify the snippet of code, you can use multiple `@@` statements to jump to the right context. For instance:

@@ class BaseClass
@@   def method():
[3 lines of pre-context]
- [old_code]
+ [new_code]
[3 lines of post-context]

The full grammar definition is below:
Patch := Begin { FileOp } End
Begin := "*** Begin Patch" NEWLINE
End := "*** End Patch" NEWLINE
FileOp := AddFile | DeleteFile | UpdateFile
AddFile := "*** Add File: " path NEWLINE { "+" line NEWLINE }
DeleteFile := "*** Delete File: " path NEWLINE
UpdateFile := "*** Update File: " path NEWLINE [ MoveTo ] { Hunk }
MoveTo := "*** Move to: " newPath NEWLINE
Hunk := "@@" [ header ] NEWLINE { HunkLine } [ "*** End of File" NEWLINE ]
HunkLine := (" " | "-" | "+") text NEWLINE

Example patch:

```
*** Begin Patch
*** Add File: hello.txt
+Hello world
*** Update File: src/app.py
*** Move to: src/main.py
@@ def greet():
-print("Hi")
+print("Hello, world!")
*** Delete File: obsolete.txt
*** End Patch
```

# Token Budget Discipline (SOFT — read this before composing the patch)

A single `apply_patch` call is one tool message. If it is large and the provider hits a rate limit, throttles, or stalls mid-stream, the **entire payload must be re-transmitted** on retry — wasted tokens, wasted minutes, and your session context burns down the same. Cache stays warmer when each call is smaller, too.

Soft budget per call:
- **≤ 5 hunks**, OR **≤ ~200 changed lines** (whichever comes first)
- One coherent semantic unit per call (e.g. "rename X in file A", "extract helper in file B") — not "refactor module"

For larger work, **split across multiple `apply_patch` calls**:
1. Plan the change set first (which files, which logical steps).
2. Apply one logical step per call. Read → patch → verify → next.
3. If you find yourself composing a patch with 10+ hunks or 500+ lines, stop and split it.

Exception: pure file creation (`*** Add File`) of a single new file can exceed these limits, since there is no match-failure risk and no benefit to splitting a brand-new file.

# Best Practices & Error Recovery

- You must include a header with your intended action (Add/Delete/Update)
- You must prefix new lines with `+` even when creating a new file
- File references can only be relative, NEVER ABSOLUTE.
- Context Lines (`@@`): The string after `@@ ` is used to find the exact location in the file. It MUST be unique and it MUST exactly match the source code snippet (including indentation, but excluding the line number prefix printed by the `read` tool).
- Exact Matches (`-`): Lines starting with `-` must precisely match the file's current whitespace and text.
- Indentation (`+`): Lines starting with `+` will be inserted exactly as written (minus the `+` character). Ensure their indentation aligns with the surrounding code.
- Handling Failures: If you receive `verification failed: Error: Failed to find expected lines` or `Cannot apply patch`, it means your `@@` context or `-` lines did not match the file. Do not guess or retry blindly. Immediately call the `read` tool to view the file's current exact content, then create a new patch using the exact spacing you just read.
