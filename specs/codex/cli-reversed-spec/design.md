# Design: codex/cli-reversed-spec

## Context

This spec is a documentation-product, not an implementation-product. It exists because OpenCode's codex provider work keeps re-deriving the same upstream behavioural facts from greps, which is slow, context-heavy, and error-prone (cache-4608 anchor error, May 11). The deliverable is a rigorously-audited reference that downstream specs cite instead.

## Goals / Non-Goals

### Goals
- One chapter per wire-affecting subsystem, IDEF0-decomposed, code-anchored, audited.
- Drift-guarded via `wiki_validate` so submodule bumps surface anchor breakage.
- OpenCode delta map per chapter for fast triage.
- Audit protocol that produces independently-verifiable evidence.

### Non-Goals
- Internal subsystems unrelated to wire shape (sandboxing, TUI rendering, etc.).
- Code change in any package.
- Anthropic / Google / other-provider analogues.
- Performance benchmarking.

## Chapter authoring workflow (per chapter)

```
draft → claim-extract → anchor-each-claim → audit-pass → record-event → promote
                                              ↑
                                              if mismatch found, back to draft
```

Each chapter is one batch. We do not start chapter N+1 until chapter N's audit passes. This avoids the cache-4608 failure mode of layering hypotheses on unverified foundations.

## c4.json scope clarification (2026-05-11)

`c4.json` exists at the **root of this spec** to depict how the reference relates to upstream codex-cli, OpenCode codex provider, and downstream provider specs. **It is NOT part of the per-chapter template.** C4's component/container/code layers overlap with the chapter's Module Architecture (block + stack) diagrams; carrying a separate c4 artifact per chapter would be redundant for an AI-reproduction / reverse-engineering workflow. Chapters use Module Architecture + IDEF0 + GRAFCET + Datasheets only — see `miatdiagram` skill §4.1 for the deprecation rationale.

## Audit protocol (canonical)

1. **Claim extraction**: every factual statement in the chapter body listed as `Cn: <statement>` in the audit file.
2. **Anchor verification**: for each `Cn`, the audit file records the corresponding `spec_add_code_anchor` entry's file/line/symbol and an excerpt that supports the claim. Mismatch = audit-failed.
3. **Cross-check sample**: at least one anchor per chapter targets a TEST or TYPE definition. Pure comment-only anchors do not count.
4. **Submodule pin**: `git -C refs/codex rev-parse HEAD` captured.
5. **Open questions**: claims that could not be verified appear as a `Q1, Q2, ...` list in the chapter body AND audit file. Better to leave a gap than fabricate a citation.
6. **Sign-off event**: `spec_record_event` with summary `audit pass chapter-NN: M claims / M anchors / SHA <sha> / K open questions` and body listing the audit table.

## Decisions

<!-- DD entries appended via spec_record_decision -->
- **DD-1**: Reference submodule SHA pinned at framework promotion: `refs/codex` HEAD = `76845d716b720ca701b2c91fec75431532e66c74` (rust-v0.0.2504301132-6092-g76845d716b). Every chapter audit re-pins SHA at audit time. Submodule advances re-open affected chapters via wiki_validate drift_code_anchors.
- **DD-2**: Chapter ordering: 01 Entry-points → 02 Auth+Identity → 03 Session/Turn → 04 Context fragments → 05 Tools/MCP → 06 Responses request build → 07 HTTP SSE → 08 WebSocket → 09 Compact → 10 Subagents → 11 Cache/prefix → 12 Rollout/Telemetry. Why: dependency order. 01-03 establishes the substrate (process, identity, session state). 04-05 supply request content. 06 assembles. 07-09 ship. 10 specialises for subagent variants. 11 explains observable cache behaviour. 12 closes with peripheral surfaces. No chapter forward-references a not-yet-audited chapter.
- **DD-3**: Audit gate: a chapter fails audit if (a) any claim lacks a code anchor, (b) any anchor's file:line doesn't actually contain content supporting the claim, OR (c) zero anchors point to TEST or TYPE definitions (only comments cited). Why: comments rot; tests and types are compiled/executed contracts. Cross-check requirement forces at least one rooted-in-compiled-code citation per chapter so the audit isn't performative.
- **DD-4**: Chapter content lives in dedicated files under `chapters/` (e.g. `chapters/01-entry-points.md`, `chapters/idef0.01.json`). Audit evidence under `audit/` (`audit/chapter-01-audit.md`). Why: keeps the root spec artifacts (proposal/spec/design) reviewable in one read; chapter detail is opt-in. Also gives `wiki_validate` a clean path-based way to enumerate audit coverage.
- **DD-5**: Downstream consumption pattern: every codex-provider spec citing this reference uses `chapters/<NN>-<slug>.md#<section>` form. spec_amend on downstream specs is triggered if their cited chapter changes (drift-suspect status propagates downstream by convention, even though plan-builder does not yet enforce it automatically). Why: separate the "what upstream does" knowledge from the "what we should do" decisions; downstream specs stay short and decision-focused.
